> ### ℹ️ About the data — please read first
> This repository runs on a **synthetic dataset of 70,000 bookings** generated by a reproducible script ([`src/data_generation/generate_synthetic_data.py`](src/data_generation/generate_synthetic_data.py)). It is **not** real customer data.
>
> The schema, value ranges, and business patterns (booking-source mix, ~92% deluxe-room preference, ~35% cancellation rate, seasonality, ~11% repeat guests) are modeled on a **real hotel analytics group project** I worked on in graduate school, where the original data was small (~2,000 bookings) and confidential. This repo recreates those dynamics **at scale** to demonstrate an end-to-end ML pipeline.
>
> **Every metric below is the real output of the models on this synthetic data** — nothing is hand-set. Because the data is synthetic, the *patterns* illustrate methodology, not findings about any real hotel.

## 📌 What this project does

A full, reproducible pipeline that takes raw hotel-booking records and answers three operational questions:

1. **Will a booking be cancelled?** — binary classification benchmarked across 7 algorithms
2. **What revenue will a booking generate, and what will the hotel earn next quarter?** — regression + Prophet time-series forecasting
3. **Which guests matter most?** — K-Means segmentation (with PCA visualisation) and revenue-concentration analysis

**Models benchmarked:** 9 classifiers (Logistic Regression, Naive Bayes, KNN, Decision Tree, Linear SVM, Random Forest, Gradient Boosting, LightGBM, XGBoost), 6 regressors (Linear, Ridge, Lasso, Random Forest, LightGBM, XGBoost), Prophet forecasting, and K-Means + PCA.

📊 **Interactive dashboard:** see [`dashboard/index.html`](dashboard/index.html) (self-contained, hosts on GitHub Pages) plus Tableau/Power BI-ready extracts in [`dashboard/tableau_ready/`](dashboard/tableau_ready).

Run the whole thing with one command:

```bash
pip install -r requirements.txt
python -m src.data_generation.generate_synthetic_data --rows 70000   # build data
python run_pipeline.py                                               # run everything
python -m src.evaluation.build_dashboard_data                        # dashboard aggregates
python -m src.evaluation.build_dashboard_html                        # build dashboard/index.html
```

## 📊 Results (real model outputs on the synthetic data)

### Cancellation prediction — 9-model benchmark

| Model | AUC | Accuracy | F1 |
|-------|-----|----------|----|
| **Gradient Boosting** | **0.729** | 0.698 | 0.46 |
| Logistic Regression | 0.726 | 0.690 | 0.51 |
| Linear SVM | 0.726 | 0.692 | 0.51 |
| XGBoost | 0.725 | 0.696 | 0.47 |
| LightGBM | 0.720 | 0.692 | 0.47 |
| Decision Tree | 0.716 | 0.690 | 0.44 |
| Naive Bayes | 0.699 | 0.677 | 0.52 |
| KNN (k=15) | 0.693 | 0.679 | 0.47 |
| Random Forest | 0.692 | 0.675 | 0.47 |

Gradient boosting leads; **XGBoost is selected as the production model** (within 0.004 AUC of the top, best on the revenue task, fast inference, native feature importance). Top cancellation drivers: **OTA booking source, lead time, prepaid payment**. Cancellation is only partially predictable from booking metadata — an honest ceiling that mirrors the original project's conclusion.

### Revenue regression — predicting booking value

| Model | R² | RMSE (₹) | MAE (₹) |
|-------|-----|----------|---------|
| **XGBoost** | **0.967** | **1,010** | 671 |
| LightGBM | 0.967 | 1,014 | 674 |
| Random Forest | 0.962 | 1,096 | 720 |
| Linear / Ridge / Lasso | 0.947 | 1,292 | 827 |

Tree ensembles beat linear models by learning nonlinear pricing interactions (room type × season × length of stay). *(The mechanically-derived per-night rate is deliberately excluded to avoid target leakage.)*

### Revenue forecasting — Prophet vs. baseline

| Metric | Prophet | Seasonal-naive baseline |
|--------|---------|-------------------------|
| RMSE | 97,115 | 145,930 |
| MAPE | **15.5%** | — |

**Prophet reduces forecast error by ~33.5%** over a seasonal-naive baseline (90-day horizon, time-series cross-validation), capturing weekly and yearly seasonality.

### Customer segmentation & revenue concentration

K-Means surfaces four guest segments, including a **high-value segment (~14% of guests, ~₹16,700 avg lifetime revenue)**. Pareto analysis: the **top 18% of guests account for ~52% of revenue** — the actionable retention target.

## 🔬 Methodology

**Feature engineering** — datetime decomposition (month, weekday, weekend, quarter), behavioural features (repeat-guest sequence, hot-room flag, parsed adult/child counts), one-hot encoding of categoricals.

**Modelling** — stratified train/test split; 7-model classification benchmark and 6-model regression benchmark under identical splits; trees on raw features, linear/distance models on scaled features.

**Forecasting** — daily revenue aggregated from honoured bookings; Prophet with weekly + yearly seasonality, evaluated by rolling-origin cross-validation against a seasonal-naive baseline.

**Honesty notes baked into the code** — target leakage is explicitly removed from the revenue model; the synthetic generator documents exactly which relationships it encodes, so results are interpretable and reproducible (`--seed`).
