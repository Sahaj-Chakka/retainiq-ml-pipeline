> ### ℹ️ About the data — please read first
> This repository runs on a **synthetic dataset of 70,000 bookings** (37 columns) generated by a reproducible script ([`src/data_generation/generate_synthetic_data.py`](src/data_generation/generate_synthetic_data.py)). It is **not** real customer data.
>
> The schema and business patterns (booking-source mix, ~92% deluxe preference, ~30% cancellation rate, seasonality, ~11% repeat guests) are modeled on a **real hotel analytics group project** I did in graduate school, where the original data was small (~2,000 bookings) and confidential. This repo recreates those dynamics **at scale** to demonstrate a full analytics workflow. The original R analysis is preserved in [`R/`](R/).
>
> **Every metric below is the real output of the models on this synthetic data** — nothing is hand-set. Because the data is synthetic, the *patterns* illustrate methodology, not findings about any real hotel.

## 📌 Analytical approach

This project follows the **standard, defensible analytical sequence** — statistics first, machine learning second:

1. **Logistic regression first** → identify which variables significantly predict cancellation, in which direction, and how confidently (coefficients, odds ratios, p-values)
2. **Diagnostics** → multicollinearity (VIF), model fit (pseudo-R², AIC), threshold optimisation
3. **Parsimonious model** → keep only significant, non-collinear variables
4. **Then ML** → tree/boosting models on the *same significant feature set*, to see whether nonlinear methods beat the linear baseline
5. **Then business analyses** → revenue forecasting, RFM segmentation, lead-time/deposit policy, dynamic pricing, overbooking — each anchored to what the regression established

This ordering means every modelling choice is *justified by evidence*, not intuition.

📊 **Interactive dashboard:** [`dashboard/index.html`](dashboard/index.html) — self-contained, hosts free on GitHub Pages. Tableau/Power BI-ready extracts in [`dashboard/tableau_ready/`](dashboard/tableau_ready).

## 1 · Logistic regression — what drives cancellation?

A full logit model on all candidate variables (Pseudo-R² 0.274, McFadden) surfaced **11 statistically significant predictors** (p < 0.05). VIF flagged `Avg_Price_Per_Room` (VIF 23.6) and `N_Adults` (VIF 9.0) as collinear — both excluded. The **parsimonious model retains the same Pseudo-R² (0.2739)** with far fewer variables, confirming the dropped features added noise, not signal.

**Significant cancellation drivers (by odds ratio):**

| Variable | Odds Ratio | Effect on cancellation |
|----------|-----------|------------------------|
| Previous Cancellations | **1.52×** | Each prior cancellation raises re-cancel odds 52% |
| No Meal Plan | 1.38× | Booking without meals → 38% higher odds |
| Web booking (vs OTA) | 1.12× | Slightly higher than OTA |
| Lead Time (per day) | 1.04× | Compounds — 60-day lead ≈ 2.5× a same-week booking |
| Special Requests (per request) | **0.91×** | More requests → more invested → less likely to cancel |
| Peak month | 0.83× | High-demand months cancel less |
| Travel Agent (vs OTA) | 0.49× | Agents have skin in the game |
| Pay-at-Hotel (vs Prepaid) | **0.35×** | Prepaid OTA bookings are costless to cancel |
| Corporate (vs OTA) | 0.30× | Lowest-risk *and* highest-revenue segment |
| Direct / Front Desk (vs OTA) | **0.27×** | Direct bookings cancel least |

**Threshold tuning:** at the default 0.50 cutoff F1 is 0.61; lowering to **0.30 lifts F1 to 0.656 with 80% recall** — the right trade-off for a hotel, where missing a cancellation costs more than a false alarm. (Validation AUC 0.840.)

## 2 · Cancellation ML benchmark — do nonlinear models beat the logit?

9 classifiers on the significant feature set, identical splits:

| Model | AUC | Accuracy | F1 |
|-------|-----|----------|-----|
| **Gradient Boosting** | **0.846** | 0.793 | 0.60 |
| Logistic Regression | 0.840 | 0.787 | 0.62 |
| XGBoost | 0.840 | 0.789 | 0.60 |
| LightGBM | 0.839 | 0.787 | 0.60 |
| Linear SVM | 0.839 | 0.785 | 0.61 |
| Random Forest | 0.838 | 0.787 | 0.59 |
| Decision Tree | 0.833 | 0.785 | 0.58 |
| Naive Bayes | 0.814 | 0.715 | 0.63 |
| KNN (k=15) | 0.810 | 0.763 | 0.55 |

**Key finding:** gradient boosting beats logistic regression by only **0.006 AUC**. When tree ensembles barely edge out a linear model, the relationships are **largely linear and additive** — an honest, senior-level conclusion. XGBoost is selected for production (≈ top AUC, wins the revenue task, native SHAP support, fast inference).

> **Note on AUC vs F1:** Gradient Boosting leads on AUC (ranking quality) while Logistic Regression and Naive Bayes show higher F1 at the default threshold. This is expected — AUC is threshold-independent; F1 isn't. Production deployment uses the tuned 0.30 threshold (see §1).

## 3 · Revenue regression — predicting booking value

6 regressors on honoured bookings (per-night rate excluded to prevent target leakage):

| Model | R² | RMSE (₹) | MAE (₹) |
|-------|-----|----------|---------|
| **XGBoost** | **0.963** | **1,215** | 789 |
| LightGBM | 0.962 | 1,223 | 793 |
| Random Forest | 0.955 | 1,327 | 862 |
| Linear / Ridge / Lasso | 0.937 | 1,579 | 1,037 |

Tree ensembles beat linear models by ~0.026 R² — they capture nonlinear pricing interactions (room type × season × length of stay) that the linear models can't.

## 5 · Guest segmentation — RFM tiers with actions

RFM (Recency · Frequency · Monetary) scoring → five named, actionable tiers:

| Tier | % Guests | % Revenue | Recommended action |
|------|----------|-----------|---------------------|
| **VIP** | 11.0% | 20.6% | Loyalty reward + early access to offers |
| At Risk | 22.9% | 22.4% | Proactive win-back within 30 days |
| New | 11.4% | 10.6% | Welcome sequence + 2nd-stay incentive |
| Loyal | 15.5% | 10.9% | Targeted upsell (upgrade, meal plan) |
| Lapsed | 39.2% | 35.5% | Reactivation offer or retire from CRM |

Pareto: the **top 18% of guests drive ~50% of revenue**. (A K-Means + PCA clustering is also included in [`src/segmentation/`](src/segmentation).)

## 6 · Revenue-management analyses

**Lead-time & deposit policy** — cancellation rate climbs from **11% (0–7 days) to 76% (60+ days)**. A non-refundable-deposit policy for OTA/Web bookings with 30+ day lead time targets ~10,000 high-risk bookings.

**Dynamic pricing / RevPAR** — RevPAR decomposed into ADR × Occupancy by month; gross-vs-net revenue per source quantifies the true cost of OTA commission (15–20%).

**Overbooking (Monte Carlo, 10k sims)** — with a 14-room inventory and ~30% cancellation, the optimal policy is to **overbook by 1 room** (walk-out probability 0.6%, expected net ₹35,886/night), with seasonal sensitivity analysis for peak vs off-peak.
